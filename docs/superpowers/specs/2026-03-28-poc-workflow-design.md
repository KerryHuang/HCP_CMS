# POC 驗證流程設計

## 背景

目前開發新功能時，從設計到實作之間缺少「快速驗證假設」的階段，導致：
- 技術可行性問題（API 行為與預期不同、效能不如預期）在實作後才發現
- 需求理解問題（邏輯不對、缺少邊界情況）在測試中才浮現
- 來回修改多次，開發效率低

## 方案

新增 `/poc` 技能 + writing-plans 風險標記規則，雙軌並行。

---

## 1. `/poc` 技能

### 觸發方式
- 手動：使用者輸入 `/poc`
- 自動：執行 writing-plans 產出的 `[POC]` 標記步驟時

### 流程（6 步）

1. **定義假設** — 明確寫出要驗證什麼（例：「PySide6 的 QTreeWidget 支援 drag & drop 排序」）
2. **判斷驗證方式**
   - 技術可行性 → 寫拋棄式原型碼到 `_poc/`
   - 需求理解 → 列出邊界情境 + 預期行為，逐項確認
3. **執行驗證** — 在 `_poc/` 目錄寫最小原型，可直接 import 專案模組來測試
4. **記錄結論** — 在 `_poc/` 內建立 `FINDINGS.md`，記錄：驗證了什麼、結果、對正式實作的影響
5. **回饋到計畫** — 根據結論調整實作計畫（修改步驟、新增邊界測試案例等）
6. **清理** — 詢問使用者後刪除 `_poc/` 內容

### 關鍵原則
- 原型碼不需要遵守 TDD、lint 等品質要求（是拋棄式的）
- 正式實作仍然走完整 TDD 流程
- `_poc/` 已被 `.gitignore` 排除，不會進版控

---

## 2. writing-plans 風險標記規則

### 風險評估維度

| 維度 | 高風險信號 |
|------|-----------|
| **技術可行性** | 首次使用的 API、複雜 SQL 查詢、跨層互動、第三方服務整合、效能敏感操作 |
| **需求確定性** | 模糊的邊界條件、多種可能的 UI 行為、未確認的業務規則、複雜的狀態轉換 |

### 標記方式
- 任一維度為高風險 → 該步驟標記 `[POC]`
- 標記時附註風險原因，例如：`[POC: 首次使用 QThread + Signal 跨線程通訊]`
- `[POC]` 步驟在正式實作前，必須先用 `/poc` 流程驗證
- 如果所有步驟都低風險，不需要標記任何 `[POC]`

### 範圍
- 此規則只影響計畫產出階段，不影響已有的 TDD、lint 等品質流程
