**二、系統背景**

**2.1 專案背景**

HCP（Human Capital Plus）系統為企業人力資源管理軟體，其客服團隊（CS）每日需處理來自多家客戶的技術詢問、異常回報與客制開發需求。過往客服人員以人工方式讀取 Outlook .msg 信件、判斷問題類型、手動填寫 Excel 追蹤表，工作流程繁瑣且易出現漏記、分類不一致等問題。

**2.2 現況問題與痛點**

-   信件量大，人工讀取與分類耗時，易遺漏

-   多家客戶共用同一信箱，公司判別依賴人工

-   問題類型分類標準不統一，導致月報統計不準確

-   Excel 追蹤表需手動維護，版本管理困難

-   Mantis Bug Tracker 與客服案件無法自動關聯，需雙重紀錄

-   QA 知識庫散落各處，無統一索引與搜尋機制

-   月報需手動整理，費時費力

**2.3 系統目標**

-   自動讀取 .msg 信件，解析寄件人、主旨、信件內容

-   依規則自動判斷系統產品、問題類型、錯誤類型、優先級

-   建立 SQLite 案件資料庫，取代 Excel 手動追蹤

-   自動產生 Excel 追蹤表與月報，供管理層檢視

-   與 Mantis Bug Tracker 雙向同步，避免重複輸入

-   建立 QA 知識庫，累積常見問題解答

-   提供瀏覽器介面（HTML），讓客服人員查詢、下載報表

**F04 --- 資料匿名化**

在寫入 QA 知識庫前，對問題描述與解答進行個資移除，避免洩露客戶姓名及內部員工資訊。共 16 條匿名化規則：

-   ① email → \[email\]　② URL → \[URL\]　③ IP → \[IP\]

-   ④ 稱謂+姓名（您好 XXX、Dear XXX）→「您好」

-   ⑤ 寄件人簽名行（From: / 寄件人：）→ 略過

-   ⑥ 完整姓名+公司（XXX 先生/小姐 from YYY）→「相關人員」

-   ⑦ 客戶公司英文域名 → 「貴客戶」

-   ⑧ CS 人員識別詞（from HCP、by 客服）→ 略

-   ⑨ Hi/Hello \[英文名\] → Hi

-   ⑩ 敬啟者/致 → 略

-   ⑪ Best regards/Thanks + 姓名 → 略

-   ⑫ 職稱+中文姓名（工程師 XXX、承辦人 XXX）→「相關人員」

-   ⑬ 姓名\|職稱格式（簽名欄）→「（簽名已略）」

-   ⑭ 獨立英文人名（單行）→「相關人員」

-   ⑮ 公司中文別名（COMPANY\_ALIAS 映射後的中文名）→「貴客戶」

-   ⑯ 獨立 2-4 個中文字（單行姓名）→「相關人員」

**F10 --- 多輪對話串追蹤（Thread Tracking）**

同一問題的多封往來信件應歸屬同一對話串，透過以下機制關聯：

-   比對依據：① Mantis 票號相同；② 同公司 + 主旨相似（去除 RE:/FW: 後比對）

-   根案件（root case）：linked\_case\_id 為空，為對話串第一封信件對應的案件

-   子案件：linked\_case\_id 設為根案件 ID，reply\_count 在根案件累計

-   重新開啟：若根案件已「已回覆」但客戶再次來信，狀態回到「處理中」，progress 追加說明

-   來回次數顯示：Excel 追蹤表中的「來回次數」欄位動態計算整串對話次數

**五、非功能需求**

**5.1 效能需求**

-   單次信件處理（含 DB 寫入與 JS 匯出）：50 封信件 \< 30 秒

-   Excel 追蹤表產生（100 筆案件）：\< 5 秒

-   月報產生（12 個月資料）：\< 10 秒

**5.2 可靠性需求**

-   已處理信件不得重複建案（processed\_files 機制）

-   信件處理失敗不影響其他信件（逐筆 try-except，錯誤記錄後繼續）

-   Excel 報表寫入失敗（檔案被鎖定）需有明確錯誤訊息

**5.3 可維護性需求**

-   分類規則集中於 rules\_config.py，與業務邏輯完全分離

-   新增客戶只需修改 COMPANY\_ALIAS，無需改動其他代碼

-   問題類型/錯誤類型新增只需在對應 list 加入 tuple，無需修改判斷邏輯

**5.4 可攜性需求**

-   支援 Windows 10/11，不依賴特定 Outlook 版本（僅讀取 .msg 檔）

-   可打包為 .exe 供無 Python 環境的電腦使用

-   HTML 介面單一檔案，開啟即用，無需 Web Server

**5.5 安全性需求**

-   個人資料（姓名、Email、電話）在寫入 QA 前自動匿名化

-   DB 儲存於本機（cs\_tracker.db），不上傳至外部服務

-   Mantis SOAP 登入帳密存於 user\_config.json，不硬寫於代碼中

**八、重構前待確認事項**

**8.1 架構決策**

-   **是否維持 SQLite 單機 DB 或改用 PostgreSQL/MySQL（多人協作需求？）**

-   **HTML 單檔介面是否改為 Web App（需 FastAPI/Flask backend？）**

-   **是否引入自動化測試（pytest + 模擬信件）**

-   **打包方式：PyInstaller exe vs. Docker vs. 安裝包**

**8.2 功能範圍**

-   **規則引擎是否改為資料驅動（DB/JSON 儲存規則，不再是 .py 硬寫）**

-   **是否支援 .eml 格式（目前僅支援 .msg）**

-   **是否需要 email 直連（IMAP/Exchange API，不需手動匯出 .msg）**

-   **Mantis SOAP API 是否需要升級為 REST API（若 Mantis 版本支援）**

-   **QA 知識庫是否需要全文搜尋（FTS5 或 Elasticsearch）**

-   email\_processor.py 單一檔案超過 3000 行，需要按責任拆分為多個模組

-   規則設定有三個入口（rules\_config.py / rules\_editor.py / HTML RulesEditorTab），可能不一致
